# REST API Гарын Авлага

---

## 1. REST Архитектурын Зарчим

REST (Representational State Transfer) нь Roy Fielding 2000 онд өөрийн докторын ажилд тодорхойлсон архитектурын хэв маяг юм. REST нь дараах 6 үндсэн зарчмаас бүрдэнэ.

---

### 1.1 Client–Server (Үйлчлүүлэгч–Сервер тусгаарлал)

Үйлчлүүлэгч (UI/frontend) болон сервер (data/backend) нь бие биеэсээ тусаарч, зөвхөн тодорхойлсон интерфейсээр харилцана.

**Давуу тал:**
- Нэг талыг өөрчлөхөд нөгөө тал нөлөөлөхгүй
- Frontend болон backend бие даан хөгжих боломжтой

---

### 1.2 Stateless (Төлөвгүй байдал)

Сервер нь үйлчлүүлэгчийн өмнөх хүсэлтийн мэдээллийг хадгалдаггүй. Хүсэлт бүр өөрөө шаардлагатай бүх мэдээллийг агуулсан байна.

**Зөв:**
```
GET /orders/123
Authorization: Bearer <token>
```

**Буруу:**
```
GET /orders/123
(сессид нэвтэрсэн гэж тооцох)
```

---

### 1.3 Cacheable (Кэшлэх боломж)

Хариуг кэшлэж болох эсэхийг тодорхой зааж өгнө. Энэ нь гүйцэтгэл болон хэмнэлтийг сайжруулдаг.

**HTTP Header жишээ:**
```
Cache-Control: max-age=3600
ETag: "abc123"
```

---

### 1.4 Uniform Interface (Нэгдмэл интерфейс)

API нь нэгдмэл, тогтвортой интерфейстэй байна. Дөрвөн дэд зарчим:

| Дэд зарчим | Тайлбар |
|---|---|
| Resource identification | Нөөц нь URL-ээр тодорхойлогдоно |
| Manipulation through representations | JSON/XML-ээр нөөцийг удирдана |
| Self-descriptive messages | Хүсэлт өөрийгөө тайлбарласан байна |
| HATEOAS | Хариуд дараагийн боломжит үйлдлүүдийн холбоос орно |

---

### 1.5 Layered System (Давхаргат систем)

Үйлчлүүлэгч нь шууд сервертэй холбогдсон эсэхийг мэдэхгүй. Завсарт load balancer, cache, security gateway байж болно.

```
Үйлчлүүлэгч → CDN → Load Balancer → API Gateway → Сервер
```

---

### 1.6 Code on Demand (Шаардлагаар код) — Заавал биш

Сервер нь үйлчлүүлэгч рүү гүйцэтгэгдэх код (JavaScript гэх мэт) илгээж болно. Энэ зарчим заавал биш (optional).

---

## 2. HTTP Method ба Статус Код

### 2.1 HTTP Methods (Аргууд)

| Method | Үйлдэл | Idempotent | Safe | Body |
|---|---|---|---|---|
| GET | Унших | Тийм | Тийм | Үгүй |
| POST | Үүсгэх | Үгүй | Үгүй | Тийм |
| PUT | Бүтнээр шинэчлэх | Тийм | Үгүй | Тийм |
| PATCH | Хэсэгчлэн шинэчлэх | Үгүй* | Үгүй | Тийм |
| DELETE | Устгах | Тийм | Үгүй | Заримдаа |
| HEAD | GET-тэй адил, биегүй хариу | Тийм | Тийм | Үгүй |
| OPTIONS | Дэмжигдэх method-ийг асуух | Тийм | Тийм | Үгүй |

> **Idempotent** — нэг болон олон удаа дуудахад үр дүн нэг байна  
> **Safe** — серверийн өгөгдлийг өөрчилдөггүй

---

### 2.2 Method-ийн Хэрэглээний Жишээ

```
GET    /users          → Бүх хэрэглэгчдийн жагсаалт
GET    /users/42       → 42-р хэрэглэгчийн мэдээлэл
POST   /users          → Шинэ хэрэглэгч үүсгэх
PUT    /users/42       → 42-р хэрэглэгчийг бүтнээр солих
PATCH  /users/42       → 42-р хэрэглэгчийн нэг талбарыг өөрчлөх
DELETE /users/42       → 42-р хэрэглэгчийг устгах
```

---

### 2.3 HTTP Статус Кодууд

#### 2xx — Амжилттай

| Код | Нэр | Хэрэглэх үе |
|---|---|---|
| 200 | OK | Хүсэлт амжилттай |
| 201 | Created | Шинэ нөөц үүслээ (POST) |
| 204 | No Content | Амжилттай, хариуны бие байхгүй (DELETE) |
| 206 | Partial Content | Хэсэгчилсэн контент |

#### 3xx — Чиглүүлэлт

| Код | Нэр | Хэрэглэх үе |
|---|---|---|
| 301 | Moved Permanently | URL байнгын шилжсэн |
| 304 | Not Modified | Кэш хүчинтэй хэвээр |

#### 4xx — Үйлчлүүлэгчийн алдаа

| Код | Нэр | Хэрэглэх үе |
|---|---|---|
| 400 | Bad Request | Буруу хүсэлт, validation алдаа |
| 401 | Unauthorized | Нэвтрээгүй |
| 403 | Forbidden | Нэвтэрсэн ч эрх байхгүй |
| 404 | Not Found | Нөөц олдсонгүй |
| 405 | Method Not Allowed | Тэр method-ийг дэмждэггүй |
| 409 | Conflict | Өгөгдлийн зөрчил (давхардсан гэх мэт) |
| 422 | Unprocessable Entity | Утгын validation алдаа |
| 429 | Too Many Requests | Rate limit хэтэрсэн |

#### 5xx — Серверийн алдаа

| Код | Нэр | Хэрэглэх үе |
|---|---|---|
| 500 | Internal Server Error | Серверийн ерөнхий алдаа |
| 502 | Bad Gateway | Upstream алдаа |
| 503 | Service Unavailable | Сервер ачааллаас гадуур |
| 504 | Gateway Timeout | Upstream timeout |

---

### 2.4 Статус Кодын Хэрэглээний Дүрмүүд

```
POST /users
→ Амжилттай бол:    201 Created  +  Location: /users/42
→ Validation алдаа:  400 Bad Request  +  алдааны тайлбар
→ Давхардсан:        409 Conflict

DELETE /users/42
→ Амжилттай бол:    204 No Content
→ Олдоогүй:         404 Not Found

GET /users/42
→ Нэвтрээгүй:       401 Unauthorized
→ Эрх байхгүй:      403 Forbidden
→ Олдоогүй:         404 Not Found
```

## 3. Resource-Based Design Гарын Авлага

### 3.1 Нөөцийн Тодорхойлолт

REST-д **нөөц (resource)** нь систем дэх аливаа нэгж юм: хэрэглэгч, захиалга, бүтээгдэхүүн гэх мэт. Нөөц бүрт өөрийн URL байна.

---

### 3.2 URL Нэрлэлтийн Дүрмүүд

| Дүрэм | Зөв | Буруу |
|---|---|---|
| Олон тооны нэрийг хэрэглэ | `/users` | `/user` |
| Жижиг үсэг хэрэглэ | `/user-profiles` | `/UserProfiles` |
| Зураас (-) хэрэглэ | `/blog-posts` | `/blog_posts` |
| Үйл үгийг бүү хэрэглэ | `/users/42` | `/getUser/42` |
| Шаталсан бүтэц | `/users/42/orders` | `/getUserOrders?id=42` |

---

### 3.3 URL Бүтцийн Загвар

```
/нөөц                        → Цуглуулга
/нөөц/:id                    → Тодорхой нэгж
/нөөц/:id/дэд-нөөц           → Харьяа цуглуулга
/нөөц/:id/дэд-нөөц/:id       → Харьяа тодорхой нэгж
```

**Бодит жишээ:**
```
GET  /products                → Бүх бүтээгдэхүүн
GET  /products/5              → 5-р бүтээгдэхүүн
GET  /products/5/reviews      → 5-р бүтээгдэхүүний сэтгэгдлүүд
POST /products/5/reviews      → Шинэ сэтгэгдэл нэмэх
DELETE /products/5/reviews/3  → 3-р сэтгэгдлийг устгах
```

---

### 3.4 Query Parameter-ийн Хэрэглэх Тохиолдлууд

Query parameter нь нөөцийн тодорхойлолтыг биш, **шүүлт, эрэмбэлэлт, хуваарь**-ыг хариуцна.

```
GET /products?category=electronics          → Шүүлт
GET /products?sort=price&order=asc          → Эрэмбэлэлт
GET /products?page=2&limit=20               → Хуваарь
GET /products?search=утас                   → Хайлт
GET /products?fields=id,name,price          → Талбар сонголт
```

---

### 3.5 Хариуны Бүтэц

#### Нэг нөөц (Object):
```json
{
  "id": 42,
  "name": "Бат",
  "email": "bat@example.com",
  "createdAt": "2024-01-15T09:30:00Z"
}
```

#### Цуглуулга (Collection):
```json
{
  "data": [
    { "id": 1, "name": "Бат" },
    { "id": 2, "name": "Дорж" }
  ],
  "meta": {
    "total": 150,
    "page": 1,
    "limit": 20,
    "totalPages": 8
  }
}
```

#### Алдааны хариу:
```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Оруулсан өгөгдөл буруу байна",
    "details": [
      { "field": "email", "message": "Email хаяг буруу форматтай байна" }
    ]
  }
}
```

---

### 3.6 Нөөцийн CRUD Жишээ — Бүрэн Загвар

Жишээ нөөц: `articles` (нийтлэл)

```
GET    /articles              → Бүх нийтлэлийн жагсаалт
POST   /articles              → Шинэ нийтлэл үүсгэх

GET    /articles/:id          → Тодорхой нийтлэл унших
PUT    /articles/:id          → Нийтлэлийг бүтнээр шинэчлэх
PATCH  /articles/:id          → Нийтлэлийн нэг талбарыг өөрчлөх
DELETE /articles/:id          → Нийтлэлийг устгах

GET    /articles/:id/comments → Нийтлэлийн сэтгэгдлүүд
POST   /articles/:id/comments → Шинэ сэтгэгдэл нэмэх
```

---

### 3.7 Нэмэлт Үйлдлүүдийг Загварчлах

Зарим үйлдэл нь нөөцийн CRUD-д яг тохирдоггүй. Тэдгээрийг дараах байдлаар загварчлана:

| Үйлдэл | Буруу | Зөв |
|---|---|---|
| Нийтлэхийг идэвхжүүлэх | POST /activateArticle | PATCH /articles/5 `{"status":"published"}` |
| Нууц үг солих | POST /changePassword | PUT /users/42/password |
| Нэвтрэх | POST /login | POST /sessions |
| Гарах | POST /logout | DELETE /sessions/:token |
| Хайлт хийх | GET /search?q=... | GET /articles?search=... |

---

### 3.8 Versioning (Хувилбарлалт)

API-г өөрчлөхөд хуучин үйлчлүүлэгчид хохирохгүй байхын тулд хувилбарлалт хэрэглэнэ.

**URL-д хувилбар:**
```
GET /v1/users
GET /v2/users
```

**Header-т хувилбар:**
```
Accept: application/vnd.myapi.v2+json
```

**Query parameter-т хувилбар:**
```
GET /users?version=2
```

> Практикт URL-д хувилбар оруулах нь хамгийн өргөн хэрэглэгддэг.

## 4. Дэд Бүтцийн Бүрэлдэхүүнүүд ба HATEOAS

### 4.1 CDN — Content Delivery Network

CDN буюу Контент Хүргэх Сүлжээ нь дэлхийн олон газарт байрлах **edge server**-уудын цуглуулга юм. Нэг үгээр хэлбэл, таны веб сервер Нью-Йоркт байгаа ч Улаанбаатарын хэрэглэгч таны зураг, CSS, JavaScript файлуудыг тэдэнд ойр байрлах серверээс татаж авна гэсэн үг.

CDN-ийн үндсэн зарчим нь маш энгийн: хэрэглэгч `/images/logo.png` файлыг хүссэн үед CDN эхлээд өөрийн ойрын edge серверийг шалгана. Хэрэв тэнд кэш байгаа бол хэдхэн миллисекундэд буцааж өгнө. Байхгүй бол origin буюу эх серверээс нэг удаа татаж авч, edge дээр хадгалаад дараагийн хэрэглэгчдэд хурдан үйлчилнэ. Энэ "эхний удаа удаан, цаашид хурдан" механизм нь том систем дэх дамжуулалтын хугацааг эрс багасгана.

CDN-д хамгийн тохиромжтой зүйл бол хэн хэрэглэсэн ч ижил хариу буцаадаг, өөрөөр хэлбэл **хэрэглэгчийн нэвтрэлтэд хамаарахгүй** контент юм: зураг, видео, font, CSS, JavaScript гэх мэт статик файлууд. Харин хэрэглэгч бүрт тусдаа хариу буцаадаг хуудас, бодит цагийн мэдээлэл, POST/PUT/DELETE хүсэлтүүд CDN-д кэшлэгдэх учиргүй бөгөөд тэдгээрийг CDN дамжуулаад origin сервер рүү шилжүүлдэг.

CDN-ийн кэшийн горимыг HTTP header-ээр удирдана. Жишээлбэл `Cache-Control: public, max-age=86400` гэвэл тухайн хариуг 24 цагийн турш CDN-д хадгалж болно гэж зааж байна. Харин `Cache-Control: private, no-store` гэвэл CDN кэшлэхгүйгээр шууд дамжуулна. `Vary: Accept-Encoding` header нь нэг URL-д агшилтын форматаар ялгаатай хувилбар кэшлэхэд хэрэглэгдэнэ, учир нь нэг хэрэглэгч gzip дэмждэг ч нөгөө нь дэмжихгүй байж болно.

```
Cache-Control: public, max-age=86400    → 1 хоногт CDN edge-д кэшлэ
Cache-Control: private, no-store        → CDN-д кэшлэхгүй, origin рүү дамжуул
Vary: Accept-Encoding                   → агшилтын форматаар ялгаж тус тусад нь кэшлэ
```

CDN нь гүйцэтгэлийн давуу талаас гадна **DDoS халдлагаас хамгаалах**, серверийн ачааллыг бууруулах, origin сервер унасан ч статик контентийг хүргэсээр байх зэрэг найдвартай байдлын давуу талтай. Cloudflare, AWS CloudFront, Fastly, Akamai зэрэг нь өнөөгийн хамгийн өргөн хэрэглэгддэг CDN үйлчилгээнүүд юм.

### 4.2 API Gateway

API Gateway нь орчин үеийн backend архитектурт **бүх хүсэлтийн нэгдсэн оролтын цэг** болдог бүрэлдэхүүн юм. Микросервисийн архитектурт хамгийн чухал үүрэг гүйцэтгэдэг бөгөөд үйлчлүүлэгч олон сервистэй шууд харьцахын оронд нэг л gateway-тэй харьцана.

API Gateway-ийн хамгийн анхдагч үүрэг бол **routing** буюу чиглүүлэлт юм. `GET /users` хүсэлт ирвэл users-сервис рүү, `GET /orders` ирвэл orders-сервис рүү дамжуулна. Үйлчлүүлэгч ард нь хэдэн сервис байгааг огт мэдэх шаардлагагүй, зөвхөн gateway-н URL-ийг л мэдэхэд хангалттай.

Хоёр дахь чухал үүрэг бол **authentication** буюу нэвтрэлтийн шалгалт. Хэрэглэгчийн JWT token эсвэл API key-г gateway дээр нэг удаа шалгахаар тохируулбал, backend сервис бүр тус тусад нь шалгах шаардлагагүй болно. Энэ нь кодын давталтыг арилгаад зогсохгүй, аюулгүй байдлын тохиргоог нэг газарт төвлөрүүлдэг.

**Rate Limiting** буюу хүсэлтийн давтамжийн хязгаарлалт мөн gateway дээр хийгддэг. Нэг IP хаягаас минутанд 100-аас илүү хүсэлт ирвэл 429 Too Many Requests буцаана гэх мэтчилэн тохируулж болно. Ингэснээр API-г хэт ашиглалт болон злоумышленникийн автомат халдлагаас хамгаална.

Gateway мөн **SSL Termination** хийдэг, өөрөөр хэлбэл гаднаас ирсэн HTTPS хүсэлтийг gateway дээр задлаад, дотоод сүлжээнд хялбар HTTP-ээр дамжуулна. Дотоод сервисүүд тус тусдаа SSL гэрчилгээ хэрэглэхгүйгээр харилцаж чаддаг болно. Гаднаас харахад бүх зүйл шифрлэгдсэн, дотооддоо хялбар, хурдан.

Нэмж дурдвал gateway нь бүх хүсэлтийг нэг газарт **логлох** боломж олгодог учир асуудал гарсан үед хаана яасныг олоход маш хялбар болно. Request/Response transformation буюу хүсэлт болон хариуны форматыг өөрчлөх боломж ч бий — жишээлбэл хуучин хэрэглэгч v1 форматаар хүсэхэд gateway v2 сервисийн хариуг v1 руу хөрвүүлж болно.

```
Үйлчлүүлэгч
     │  HTTPS хүсэлт
     ▼
┌──────────────────────────────┐
│         API Gateway          │
│  1. SSL задлах               │
│  2. Token шалгах             │
│  3. Rate limit шалгах        │
│  4. Логлох                   │
│  5. Тохирох сервис рүү чиглүүлэх │
└────────────┬─────────────────┘
             │  HTTP
      ┌──────┼──────┐
      ▼      ▼      ▼
   /users  /orders  /products
   сервис  сервис   сервис
```

AWS API Gateway, Kong, Nginx, Traefik зэрэг нь практикт хамгийн их хэрэглэгддэг хэрэгслүүд юм. Kong болон Traefik нь plugin-д суурилсан уян хатан тохиргоотой тул микросервисийн орчинд түгээмэл.

### 4.3 Load Balancer — Ачаалал Тэнцвэржүүлэгч

Load Balancer нь нэг сервис олон серверийн жагсаалт дээр ажиллаж байх үед орж ирэх трафикийг тэдгээрт жигд хуваарилах үүрэгтэй. Нэг сервер хэт дарагдаж унахаас сэргийлж, системийн найдвартай байдал болон гүйцэтгэлийг нэгэн зэрэг дээшлүүлдэг.

Хамгийн энгийн алгоритм бол **Round Robin** — хүсэлт ирэх бүрт дараалсан эргэлтээр сервер сонгоно: S1, S2, S3, S1, S2, S3... Серверүүдийн хүч ижил байх үед энэ арга сайн ажилладаг. Гэхдээ хэрэв нэг хүсэлтийн боловсруулалт удаан, нөгөөх нь хурдан байдаг бол **Least Connections** арга илүү зохистой — одоо хамгийн цөөн идэвхтэй холболттой сервер рүү чиглүүлнэ, ингэснээр аяндаа хурдан серверт илүү хүсэлт очно.

**IP Hash** арга нь хэрэглэгчийн IP хаягийг hash хийж, тухайн хэрэглэгч дандаа нэг серверт очдог болгоно. Session-д суурилсан нэвтрэлт хэрэгтэй, буюу сервер хэрэглэгчийн статусыг санах шаардлагатай тохиолдолд хэрэглэнэ. **Weighted Round Robin** нь серверүүдийн хүчийг харгалзан жин тогтоодог — 4 core-той серверт 2 core-той серверийнхээс хоёр дахин их хүсэлт оч гэх мэтчилэн.

Load Balancer-ийн маш чухал хэсэг бол **Health Check** буюу серверийн эрүүл мэндийн шалгалт. Load Balancer тогтмол хугацаанд бүх серверт жижиг хүсэлт илгээн амьд эсэхийг шалгадаг. Хэрэв сервер хариу өгөхгүй бол трафикийн жагсаалтаас автоматаар хасаж, сэргэсний дараа дахин нэмдэг. Энэ механизм ачааллын үед нэг сервер унасан ч хэрэглэгч мэдрэхгүй байх боломжийг бүрдүүлдэг.

```
Олон хэрэглэгч нэгэн зэрэг хандана
      │   │   │   │
      ▼   ▼   ▼   ▼
  ┌──────────────────┐
  │   Load Balancer  │
  │                  │
  │  [S1 ✓] [S2 ✓]  │  ← health check: S3 унасан
  │         [S3 ✗]  │     → S3 рүү хүсэлт явахгүй
  └────┬─────────┬───┘
       ▼         ▼
      S1         S2       (S3 сэргэхийг хүлээнэ)
```

Load Balancer нь **Layer 4** (сүлжээний түвшин, TCP/UDP) болон **Layer 7** (програмын түвшин, HTTP) гэж хуваагддаг. Layer 7 нь HTTP header, URL path, cookie зэрэг мэдээллийг харгалзан ухаалаг чиглүүлэлт хийж чадна — жишээлбэл `/api` хүсэлт API серверт, `/static` хүсэлт статик файлын серверт явуулна. AWS ELB, HAProxy, Nginx зэрэг нь өргөн хэрэглэгддэг хэрэгслүүд бөгөөд Nginx нь нэгэн зэрэг reverse proxy болон load balancer үүрэг гүйцэтгэж чаддагаараа онцлог.

### 4.4 Cache — Кэш

Кэш нь компьютерийн шинжлэх ухааны хамгийн хүчтэй оновчлолын аргуудын нэг бөгөөд цөм санаа нь маш энгийн: **өмнө тооцоолсон зүйлийг дахин тооцоолохгүй, хадгалсан хариуг шууд буцаа**. REST API-д кэш нь хэд хэдэн давхаргад ажилладаг.

#### Клиентийн кэш ба HTTP conditional request

Серверийн хариуд `Cache-Control: max-age=300` орсон бол браузер эсвэл апп тухайн хариуг 5 минутын турш локалд хадгалж, дахин хүсэлт илгээхгүйгээр шууд ашиглана. Гэхдээ 5 минут өнгөрсөний дараа яах вэ? Шинэ хариу заавал татах шаардлагагүй, учир нь өгөгдөл өөрчлөгдөөгүй байж болно. Энд **ETag** (entity tag) орж ирдэг.

Сервер хариуд `ETag: "v1-abc123"` header нэмэхэд энэ нь тухайн өгөгдлийн "хурууны хээ" болно. Клиент дараагийн хүсэлтэд `If-None-Match: "v1-abc123"` илгээхэд сервер өгөгдөл өөрчлөгдсөн эсэхийг шалгаад: өөрчлөгдөөгүй бол **304 Not Modified** буцаана — бие байхгүй, зөвхөн header, маш хурдан. Өөрчлөгдсөн бол шинэ ETag-тай **200 OK** буцаана.

```
[Анхны хүсэлт]
GET /products/5
→ 200 OK
   ETag: "v1-abc123"
   Cache-Control: max-age=300
   { "id": 5, "price": 9900 }

[5 минутын дараа]
GET /products/5
If-None-Match: "v1-abc123"
→ 304 Not Modified          ← өгөгдөл өөрчлөгдөөгүй, хариу бие байхгүй

[Үнэ өөрчлөгдсөн бол]
GET /products/5
If-None-Match: "v1-abc123"
→ 200 OK
   ETag: "v2-xyz789"        ← шинэ ETag
   { "id": 5, "price": 12000 }
```

#### Серверийн кэш

Серверийн талд кэш нь өгөгдлийн санд хандах дуудлагыг багасгах зорилготой. Хамгийн нийтлэг хэрэглүүр бол **Redis** — санах ойд ажилладаг, өгөгдлийн сан дуудахаасаа олон зуун дахин хурдан. Ажиллах зарчим дараах байдалтай: хүсэлт ирэхэд эхлээд Redis шалгана, байвал шууд буцаана (cache hit), байхгүй бол өгөгдлийн санаас унших (cache miss), уншсан хариуг Redis-т хадгалаад дараа дахин шаардлагатай болвол хурдан буцаана.

Кэшийн **стратеги** нь ямар тохиолдолд кэшийг шинэчлэх, хаяхыг тодорхойлдог. Хамгийн өргөн хэрэглэгддэг **Cache-Aside** (Lazy Loading) стратегид апп кэшийг өөрөө удирддаг: кэшэд байвал авна, байхгүй бол DB-ээс уншиж, кэшлэж, дараа хурдан буцаана. **Write-Through** стратегид өгөгдөл бичих үед кэш болон DB-г нэгэн зэрэг шинэчилдэг тул кэш болон DB хоёр нь үргэлж нийцтэй байдаг. **Write-Behind** стратегид эхлээд кэшт бичиж, DB-г хойш хойшлуулан шинэчилдэг учир бичих хурд маш өндөр байдаг ч сервер гэнэт унасан тохиолдолд өгөгдөл алдагдах эрсдэлтэй.

Кэшийн хамгийн хэцүү асуудал бол **cache invalidation** буюу хэзээ кэшийг хүчингүй болгох вэ гэдэг нь. Хугацааны хязгаар (TTL) тогтоовол кэш автоматаар хуучирч хаягдана — энгийн ч бодит цагт нийцэхгүй байж болно. Event-based invalidation-д өгөгдөл өөрчлөгдөх үед тухайн кэшийг тэр даруй устгадаг — нарийн ч бага зэрэг нарийн хэрэгжүүлэлт шаарддаг. Статик файлд cache busting хэрэглэдэг — URL-д хувилбарын дугаар нэмэхэд браузер шинэ файл гэж ойлгоод дахин татна: `/style.v2.css` гэх мэт.

### 4.5 HATEOAS

HATEOAS буюу "Hypermedia As The Engine Of Application State" нь REST-ийн хамгийн дээд шатны зарчим бөгөөд Roy Fielding-ийн анхны тодорхойлолтод заавал биелэгдэх нөхцөл болгон оруулсан байдаг. Гэхдээ практикт хамгийн ойлгомжгүй, хамгийн бага хэрэгжүүлэгддэг зарчим нь энэ юм.

Цөм санаа нь энгийн: **API-ийн хариу нь зөвхөн өгөгдлөөс гадна тухайн нөөцтэй холбогдон хийж болох дараагийн үйлдлүүдийн жагсаалтыг мөн агуулна**. Тэгэхэд үйлчлүүлэгч API-ийн баримт бичиг уншилгүйгээр л хариунаас чиглэл авч дараагийн алхмаа мэдэж чадна. Сайн зохион бүтээгдсэн HATEOAS API нь цэс бүхий ресторан шиг — өдөр болгон юу байгааг хэлж байдаг, хэрэглэгч цээжлэх шаардлагагүй.

Энгийн хариутай харьцуулахад ялгааг тод харуулъя. HATEOAS-гүй ердийн хариу:

```json
{
  "id": 42,
  "name": "Бат",
  "status": "active",
  "balance": 50000
}
```

Энэ хариуг хүлээн авсан үйлчлүүлэгч дараа нь юу хийж болохыг огт мэдэхгүй — API баримт бичиг нь тусад нь байдаг бөгөөд ямар URL ашиглах, ямар method дуудах, ямар эрх хэрэгтэйг тусад нь судлах ёстой. HATEOAS-тай хариу:

```json
{
  "id": 42,
  "name": "Бат",
  "status": "active",
  "balance": 50000,
  "_links": {
    "self":     { "href": "/users/42",            "method": "GET"    },
    "update":   { "href": "/users/42",            "method": "PUT"    },
    "delete":   { "href": "/users/42",            "method": "DELETE" },
    "orders":   { "href": "/users/42/orders",     "method": "GET"    },
    "deposit":  { "href": "/users/42/deposit",    "method": "POST"   },
    "suspend":  { "href": "/users/42/suspend",    "method": "POST"   }
  }
}
```

Одоо үйлчлүүлэгч хариуг уншаад л дараа нь юу хийж болохыг мэдэж авна. Нэмж хэлэхэд, хэрэв тухайн хэрэглэгч `suspend` хийх эрхгүй бол сервер тэр холбоосыг хариунд оруулахгүй байж болно — үйлчлүүлэгч туршиж үзэх ч шаардлагагүй, ямар үйлдэл хийж болохыг хариунаас шууд мэдэж байна.

Pagination буюу хуваарийн жагсаалтад HATEOAS-ийн давуу тал онцгой тод гардаг:

```json
{
  "data": [ ... ],
  "meta": { "total": 150, "page": 2, "limit": 20 },
  "_links": {
    "self":  { "href": "/products?page=2&limit=20" },
    "first": { "href": "/products?page=1&limit=20" },
    "prev":  { "href": "/products?page=1&limit=20" },
    "next":  { "href": "/products?page=3&limit=20" },
    "last":  { "href": "/products?page=8&limit=20" }
  }
}
```

Үйлчлүүлэгч нийт хэдэн хуудас байгааг тооцоолохгүйгээр, зөвхөн `next` холбоосыг дагаж л дараагийн хуудас руу очно. API-д pagination-ийн параметрийн нэр өөрчлөгдсөн ч үйлчлүүлэгчийн код өөрчлөх шаардлагагүй — учир нь тэр URL-ийг бие даан үүсгэдэггүй, серверийн буцаасан холбоосыг дагадаг.

HATEOAS-ийн **давуу тал** нь гурван зүйлд хаардаг. Нэгдүгээрт, үйлчлүүлэгч болон сервер хоёрын хамаарлыг суларгана — API-ийн URL бүтэц өөрчлөгдсөн ч хариунд байгаа холбоосыг дагасаар байвал үйлчлүүлэгч ажиллаж л чадна. Хоёрдугаарт, API нь өөрийгөө тайлбарлах чадвартай болно, тусад нь баримт бичиг байх шаардлага буурна. Гуравдугаарт, хэрэглэгчийн эрх, нөхцөлийн дагуу боломжит үйлдлийн жагсаалт динамик өөрчлөгдөнө.

**Сул тал** нь хариуны хэмжээ томордог, хэрэгжүүлэхэд нэмэлт ажил шаарддаг, ихэнх фронтэнд хөгжүүлэгч холбоосыг dynamic дагахын оронд URL-ийг hardcode хийх хандлагатай байдаг учир практикт бүрэн хэрэгжүүлэх нь хэцүү байдаг.

Өнөөдрийн практикт ихэнх компани HATEOAS-ийг бүрэн дагаагүй ч хэсэгчлэн нэвтрүүлдэг: pagination-д `_links` оруулах, нөөцийн `self` холбоосыг хариунд нэмэх нь хамгийн нийтлэг алхам бөгөөд бодит ашиг тустайн зэрэгцээ хэрэгжүүлэхэд хялбар.

### 4.6 Бүрэлдэхүүнүүдийн Харилцан Харилцаа

```
Хэрэглэгч
    │
    ▼
┌──────────┐
│   CDN    │  ← статик файл кэшлэх, газарзүйн хуваарилалт
└────┬─────┘
     │ (статик биш бол дамжуулна)
     ▼
┌──────────────┐
│  API Gateway │  ← auth, rate limit, routing, logging
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ Load Balancer│  ← трафик тарааx, health check
└───┬──────┬───┘
    ▼      ▼
  App1   App2    ← backend серверүүд
    │      │
    ▼      ▼
┌──────────────┐
│    Cache     │  ← Redis/Memcached, DB дуудлага багасгах
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  Database    │  ← бодит өгөгдөл
└──────────────┘
```

| Бүрэлдэхүүн | Гол зорилго | Байршил |
|---|---|---|
| CDN | Контентийг хэрэглэгчид ойртуулах | Хамгийн гадна |
| API Gateway | Нэгдсэн оролт, хамгаалалт | CDN-ийн ард |
| Load Balancer | Ачаалал тэнцвэржүүлэх | App-ийн өмнө |
| Cache | Хурдасгах, DB-г хамгаалах | App-ийн дотор |
| HATEOAS | API-г өөрийгөө тайлбарлуулах | API хариуны бүтэц |
