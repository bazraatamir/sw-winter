# Message Queue — Дэлгэрэнгүй Онолын Гарын Авлага

---

## 1. Асуудал: Ачааллын оргил болон хурдны зөрүү

Апп ажиллаж байхад хоёр төрлийн асуудал нийтлэг тохиолддог.

**Ачааллын оргил.** Баярын өдрүүдэд захиалга нэгэн зэрэг маш олон ирнэ. Сервер тэр бүгдийг нэгэн зэрэг боловсруулах хүчин чадалгүй бол зарим захиалга алдагдана.

**Хурдны зөрүү.** Task үүсгэх нь 10мс зарцуулдаг боловч имэйл илгээх нь 500мс зарцуулдаг. Хэрэглэгч task үүсгэхийн тулд имэйл дуустал 500мс хүлээх ёстой юу? Тийм биш.

```
Асуудал 1 — Оргил ачаалал:
100 хүсэлт/секунд ирнэ
Сервер 20 л боловсруулна
→ 80 хүсэлт алдагдана 😱

Асуудал 2 — Хурдны зөрүү:
Task үүсгэх   [10мс]
Имэйл илгээх [500мс]  ← хэрэглэгч яагаад хүлээх ёстой вэ?
→ Нийт 510мс хариу хүлээнэ 😱
```

---

## 2. Message Queue гэж юу вэ?

Message Queue бол мессежүүдийг **дарааллалсан байдлаар хадгалж, дамжуулдаг** дундын систем юм. Илгээгч (Producer) мессеж дараалалд хийнэ. Хүлээн авагч (Consumer) өөрийн хурд, хүчин чадлаараа уншиж боловсруулна.

Зүйрлэл: Банкны дугаарын систем гэж бодоорой. Харилцагч дугаар авч суунд суудаг — тэр даруй үйлчлүүлэх шаардлагагүй. Касс ажилтан чөлөөтэй болоход дараагийн дугаарыг дуудна. Ямар ч харилцагч "зогсоогүй" — бүгд дуудагдах болно, зөвхөн цагт нь.

```
Хэрэглэгч → [Task Service] → [Queue] → [Email Service]
                  ↑                           ↑
              Хурдан хариу            Өөрийн хурдаар
              буцаана (10мс)          боловсруулна (500мс)
```

---

## 3. Үндсэн Ойлголтууд

### Producer (Үүсгэгч)

Producer буюу үүсгэгч нь Queue-д мессеж илгээдэг хэсэг юм. Producer мессеж Queue-д хийгээд **хариу хүлээхгүй** — цааш үргэлжилнэ. Consumer боловсруулсан эсэх нь Producer-д хамаагүй.

### Consumer (Хэрэглэгч)

Consumer буюу хэрэглэгч нь Queue-с мессеж авч боловсруулдаг хэсэг юм. Consumer өөрийн хурд, хүчин чадлаараа мессеж уншина. Нэг Queue-г олон Consumer уншиж болно — ачаалал хуваарилагдана.

### Queue (Дараалал)

Queue бол мессежүүд хадгалагдаж байдаг дарааллын сав юм. Ихэвчлэн **FIFO (First In, First Out)** зарчмаар ажилладаг — эхэлж ороод эхэлж гарна.

```
Producer          Queue               Consumer
   │                                      │
   └──→ [msg1][msg2][msg3][msg4] ──→ ─────┘
         ↑ эхэлж орсон             ↑ эхэлж гарна
```

### Acknowledgment (Баталгаажуулалт)

Consumer мессежийг амжилттай боловсруулсны дараа Queue-д **"боловсруулсан"** гэж мэдэгддэг. Үүнийг ACK (Acknowledge) гэнэ. ACK ирэхгүй бол Queue тухайн мессежийг дахин илгээнэ.

```
Queue ──→ Consumer: мессеж авна
Consumer: боловсруулна
Consumer ──→ Queue: ACK (дууслаа)
Queue: мессеж устгана ✓

Хэрэв Consumer унавал:
Queue: ACK ирсэнгүй
Queue: мессежийг өөр Consumer-д дахин илгээнэ ✓
```

---

## 4. Message Queue-ийн Дотоод Ажиллагаа

### Мессеж Queue-д орох

Producer мессеж илгээхэд Queue:
1. Мессежийг **хадгалах санд** бичнэ (санах ой эсвэл disk)
2. Producer-д **"хүлээн авлаа"** гэж баталгаажуулна
3. Мессежийг дарааллын **сүүлд** нэмнэ

### Мессеж Queue-с гарах

Consumer мессеж хүсэхэд Queue:
1. Дарааллын **эхний мессежийг** буцаана
2. Мессежийг **"нөөцлөгдсөн"** гэж тэмдэглэнэ — бусад Consumer энэ мессежийг аваагүй байна
3. Consumer **ACK илгээхэд** мессежийг бүрмөсөн устгана

```
Queue-ийн дотоод төлв:

[нөөцлөгдсөн: msg1] [хүлээж байгаа: msg2] [msg3] [msg4]
         ↑
   Consumer боловсруулж байна
   ACK ирэхийг хүлээж байна
```

### Visibility Timeout

Мессеж Consumer-д явсны дараа тодорхой хугацаанд бусад Consumer-т "харагдахгүй" болдог. Энэ хугацааг **Visibility Timeout** гэнэ. Consumer тухайн хугацаанд ACK илгээхгүй бол мессеж дахин "харагдаж" өөр Consumer-д очно.

```
msg1 Consumer A-д яваад 30 секунд өнгөрлөө
ACK ирсэнгүй (Consumer A унасан байж болно)
→ msg1 дахин Queue-д гарч ирнэ
→ Consumer B авна
```

## 5. Queue-ийн Хэлбэрүүд

### 5.1 Point-to-Point (Цэг хоорондын)

Нэг мессежийг яг **нэг Consumer** хүлээн авна. Олон Consumer байвал ачаалал хуваарилагдах боловч нэг мессежийг зөвхөн нэг нь авна.

```
Producer ──→ [Queue] ──→ Consumer A  (эсвэл)
                    ──→ Consumer B  (эсвэл)
                    ──→ Consumer C
Нэг мессеж → нэг Consumer
```

**Хэрэглээ:** Имэйл илгээх, тайлан үүсгэх — нэг даалгаврыг нэг ажилтан гүйцэтгэх.

### 5.2 Publish-Subscribe (Нийтлэх-Бүртгэх)

Нэг мессежийг **бүх subscriber** хүлээн авна. Нэг мессеж олон Consumer-т хүрнэ.

```
Producer ──→ [Topic] ──→ Consumer A  (мөн)
                    ──→ Consumer B  (мөн)
                    ──→ Consumer C
Нэг мессеж → бүх Consumer
```

**Хэрэглээ:** Мэдэгдэл, аналитик — нэг үйл явдлыг олон систем мэдэх.

### 5.3 Dead Letter Queue (DLQ)

Consumer мессежийг боловсруулж чадахгүй (тодорхой тооны оролдлогын дараа) бол тухайн мессежийг **тусгай Queue-д** шилжүүлнэ. Хөгжүүлэгч дараа нь шалгаж, гараар боловсруулж болно.

```
Queue ──→ Consumer: боловсруулж чадахгүй (3 удаа оролдов)
                ↓
           [Dead Letter Queue]
                ↓
     Хөгжүүлэгч дараа нь шалгана
```

**Яагаад чухал вэ?** DLQ байхгүй бол боловсруулах боломжгүй мессеж Queue-д эргэлдэж Consumer-г дахин дахин залхаана.

## 6. Queue-ийн Гол Шинж Чанарууд

### Durability (Тогтвортой байдал)

Broker унасан ч мессежүүд алдагдахгүй байх чадвар. Дискэнд бичсэн мессеж сервер дахин эхлэхэд хэвээр байна.

```
Disk-д хадгалдаг Queue → Broker унасан ч мессеж хэвээр ✓
Санах ойд хадгалдаг Queue → Broker унавал мессеж алдагдана ✗
```

### Ordering (Дараалал)

Мессежийг ирсэн дарааллаар буцаах чадвар. Ихэнх Queue FIFO зарчмаар ажилладаг боловч олон Consumer байвал дараалал алдагдаж болно.

### Throughput (Дамжуулах чадвар)

Нэг секундад хэдэн мессеж боловсруулах чадвар. Kafka секундад сая мессеж дамжуулж чадна. RabbitMQ мянгаар хэмжигдэнэ.

### Latency (Хоцролт)

Мессеж Queue-д орсноос Consumer хүртэл хүрэх хугацаа. Бага latency шаардлагатай системд санах ойд хадгалдаг Queue (Redis) тохиромжтой.

## 7. Хамгийн Түгээмэл Message Queue Системүүд

### RabbitMQ

AMQP протоколд суурилсан, найдвартай, уян хатан Message Broker. Task Queue-д хамгийн тохиромжтой.

**Онцлог:**
- Exchange гэсэн завсарын давхарга — мессежийг Queue-д хуваарилах дүрэм тодорхойлно
- Олон протокол дэмждэг (AMQP, MQTT, STOMP)
- Management UI-тэй — Queue-ийн төлвийг визуалаар харах
- Мессеж нэг Consumer-т л очдог (point-to-point)

**Хэзээ сонгох:** Task Queue, даалгавар хуваарилах, retry логик чухал үед.

### Apache Kafka

Өндөр throughput-тай event streaming платформ. Мессежийг лог маягаар хадгалдаг тул Consumer дурын үеэс буцаж уншиж болно.

**Онцлог:**
- Мессеж устгагддаггүй — тодорхой хугацаанд хадгалагдана
- Consumer Group — нэг Consumer Group дотор нэг partition-г нэг Consumer уншина
- Өндөр throughput — секундад сая мессеж
- Ordered within partition — нэг partition дотор дараалал хатуу

**Хэзээ сонгох:** Аналитик, аудит лог, real-time data pipeline, Event Sourcing.

### Redis Pub/Sub ба Redis Streams

Redis-д суурилсан хөнгөн хэлбэр. Pub/Sub нь санах ойд ажилладаг тул Broker унавал мессеж алдагдана. Redis Streams нь мессежийг хадгалдаг тул найдвартай.

**Хэзээ сонгох:** Redis аль хэдийн ашиглаж байгаа, жижиг throughput, хялбар тохиргоо хэрэгтэй үед.

### AWS SQS

AWS-ийн удирдагдсан Message Queue үйлчилгээ. Тохируулах, удирдах шаардлагагүй — AWS бүгдийг хангана.

**Хэзээ сонгох:** AWS дэд бүтэц ашиглаж байгаа, тохируулах цаг байхгүй үед.

## 8. Queue-ийн Чухал Хэлбэрүүд

### Work Queue (Даалгаврын дараалал)

Хүнд даалгавруудыг дараалалд хийж, Worker-ууд боловсруулна. Нэг Worker завгүй бол өөр Worker авна.

```
Producer ──→ [Queue] ──→ Worker 1 (PDF үүсгэж байна)
                    ──→ Worker 2 (чөлөөтэй — авна)
                    ──→ Worker 3 (чөлөөтэй — авна)
```

**Хэрэглээ:** PDF үүсгэх, зураг боловсруулах, тайлан гаргах.

### Retry Queue (Дахин оролдох дараалал)

Боловсруулж чадаагүй мессежийг тодорхой хугацааны дараа дахин оролдоно. Сүлжээний түр доголдол гэх мэт хэсэгчилсэн алдаануудыг автоматаар шийддэг.

```
Queue ──→ Consumer: алдаа
               ↓
        [Retry Queue: 1 минутын дараа дахин оролд]
               ↓ (1 минут өнгөрнө)
        Queue ──→ Consumer: дахин оролдоно
               ↓ (дахин алдаа)
        [Retry Queue: 5 минутын дараа]
               ↓ (3 удаа оролдсоны дараа)
        [Dead Letter Queue]
```

### Priority Queue (Тэргүүлэх дарааллал)

Мессежүүдийг FIFO биш **ач холбогдлоор** нь эрэмбэлж боловсруулна. Яаралтай мессеж эрт боловсруулагдана.

```
[Яаралтай: msg_urgent] [Хэвийн: msg1] [Хэвийн: msg2]
         ↑
   Эхэлж боловсруулагдана
```

## 9. Producer-Consumer хэлбэр

### Pulling (Татах)

Consumer Queue-с мессеж **идэвхтэй хүсдэг** — "шинэ мессеж байна уу?" гэж асуудаг.

```
Consumer ──→ Queue: "мессеж байна уу?"
Queue ──→ Consumer: байвал өгнө, байхгүй бол хоосон
Consumer: тодорхой хугацааны дараа дахин асуудаг
```

**Давуу тал:** Consumer өөрийн хурдаар татдаг — хэт их мессеж ирж дарагдахгүй.
**Сул тал:** Мессеж байхгүй ч байнга асуудаг — нөөц зарцуулдаг (polling overhead).

### Pushing (Түлхэх)

Broker шинэ мессеж ирмэгц Consumer-д **шууд илгээдэг**.

```
Queue ──→ Consumer: шинэ мессеж байна, ав
Consumer: хүлээн авна
```

**Давуу тал:** Мессеж ирмэгц Consumer шууд авна — хоцролт бага.
**Сул тал:** Consumer боловсруулах хурдаас илүү мессеж ирвэл Consumer дарагдаж болно (backpressure хэрэгтэй).

## 10. Message Queue-ийн Давуу ба Сул Талууд

### Давуу талууд

**Ачааллын буфер.** Оргил ачааллын үед мессежүүд Queue-д хуримтлагдаж, Consumer аажмаар боловсруулна. Сервер дарагдахгүй.

**Хурдны тусгаарлалт.** Producer хурдан, Consumer удаан байж болно. Хоёр тал бие биеийн хурдад нөлөөлөхгүй.

**Найдвартай байдал.** Consumer унасан ч мессеж Queue-д хадгалагдана. Consumer сэргэхэд үргэлжлүүлнэ.

**Масштаблах.** Consumer-ийн тоог нэмснээр ачааллыг хуваарилж болно.

### Сул талууд

**Нэмэлт дэд бүтэц.** Message Broker суулгах, тохируулах, удирдах шаардлагатай.

**Eventual consistency.** Producer мессеж илгээсний дараа Consumer боловсруулах хүртэл хугацаа шаардагддаг. Шууд үр дүн харагдахгүй.

**Нарийн төвөгтэй debug.** Алдаа хаана гарсан нь тодорхойгүй болдог — Producer, Broker, эсвэл Consumer-д үү?

**Ordering хэцүү.** Олон Consumer байвал мессежийн дараалал алдагдаж болно.

## 11. Task Manager-т Message Queue Ашиглах Хэлбэр

Task Manager-т Queue-г дараах хэсгүүдэд ашиглах нь хамгийн тохиромжтой:

```
Task үүсгэгдэнэ
      ↓
[notification_queue] ──→ NotificationWorker
                          (оноогдсон хүнд мэдэгдэнэ)

Workspace архивлагдана
      ↓
[archive_queue] ──→ ArchiveWorker
                    (бүх task архивлана — удаан үйлдэл)

Тайлан хүсэгдэнэ
      ↓
[report_queue] ──→ ReportWorker
                   (PDF үүсгэнэ — хүнд үйлдэл)
```

Queue ашиглахгүй бол эдгээр үйлдлүүд бүгд хэрэглэгчийг хүлээлгэнэ. Queue ашиглавал хэрэглэгч шууд хариу авч, ажил дэвсгэрт гүйцэтгэгдэнэ.

---

## Товч дүгнэлт

Message Queue бол **ачааллыг хуваарилах, хурдны зөрүүг арилгах, найдвартай байдлыг хангах** дундын систем юм. Санах ойд байлгах зүйл:

- **FIFO** → эхэлж ороод эхэлж гарна
- **ACK** → Consumer боловсруулсны дараа баталгаажуулна, болохгүй бол дахин илгээнэ
- **DLQ** → боловсруулж чадаагүй мессеж тусдаа Queue-д хадгалагдана
- **Point-to-Point** → нэг мессеж нэг Consumer
- **Pub/Sub** → нэг мессеж бүх Consumer
- **RabbitMQ** → Task Queue, найдвартай хүргэлт
- **Kafka** → Streaming, аналитик, өндөр throughput
- **Жижиг аппт шаардлагагүй** — Broker нэмэлт хүндрэл авчирна
