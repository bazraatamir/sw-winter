# MVC Архитектурын Гарын Авлага
### Express.js + mysql2 — REST API Монолит Төсөлд Зориулсан

---

## 1. MVC гэж юу вэ?

MVC буюу Model-View-Controller нь програм хангамжийн архитектурын загвар бөгөөд кодыг үүрэг хариуцлагаар нь тусдаа давхаргад хуваадаг. Энэ загвар нь кодыг уншихад хялбар, засварлахад дөхөм, давхаргуудыг бие биеэс нь тусгаарласан байдлаар зохион байгуулах зорилготой.

REST API-д View давхарга нь JSON хариу буцаах үүрэгтэй болох тул бидний хувьд Controller, Service, Repository, Model гэсэн 4 давхарга ашигладаг. HTTP хүсэлт ирэхэд эхлээд Controller хүлээн авч, Service-г дуудна. Service нь бизнес логикийг гүйцэтгээд Repository-г дуудна. Repository нь mysql2 ашиглан өгөгдлийн санд хандаж, Model хэлбэрээр өгөгдлийг буцаана. Эцэст нь Controller JSON хариу үүсгэж клиент рүү илгээнэ.

## 2. Хавтасны Бүтэц

```
src/
├── config/
│   └── db.js
├── controllers/
│   ├── auth.controller.js
│   └── product.controller.js
├── services/
│   ├── auth.service.js
│   └── product.service.js
├── repositories/
│   ├── user.repository.js
│   └── product.repository.js
├── models/
│   └── product.model.js
├── routes/
│   ├── auth.routes.js
│   └── product.routes.js
├── middlewares/
│   ├── auth.middleware.js
│   ├── validate.middleware.js
│   └── errorHandler.js
├── utils/
│   └── apiResponse.js
├── app.js
└── server.js
```

config хавтас нь өгөгдлийн сангийн холболтын тохиргоог агуулна. controllers хавтас нь HTTP хүсэлт хариуцах файлуудыг агуулна. services хавтас нь бизнес логикийн файлуудыг агуулна. repositories хавтас нь өгөгдлийн санд хандах файлуудыг агуулна. models хавтас нь өгөгдлийн бүтцийн файлуудыг агуулна. routes хавтас нь endpoint-уудын тодорхойлолтыг агуулна. middlewares хавтас нь хүсэлт боловсруулах завсрын функцуудыг агуулна. utils хавтас нь нийтлэг хэрэгслүүдийг агуулна.

## 3. Давхаргуудын Дэлгэрэнгүй Тайлбар

### 3.1 Model (Загвар)

Model нь өгөгдлийн бүтцийг тодорхойлох давхарга юм. Express.js-т Sequelize зэрэг ORM ашиглахгүй бол Model нь өгөгдлийн сангийн хүснэгтийн бүтцийг JavaScript класс хэлбэрээр илэрхийлнэ. Өөрөөр хэлбэл өгөгдлийн санаас ирсэн мөр бүрийг Model объект болгон хувиргаж, тухайн объект ямар талбартай байхыг тодорхойлдог.

Model-ийн гол үүрэг нь хоёр зүйл байдаг. Нэгдүгээрт, өгөгдлийн санаас ирсэн snake_case талбаруудыг (жишээ нь category_id, created_at) JavaScript-ийн camelCase хэлбэрт (categoryId, createdAt) хөрвүүлнэ. Хоёрдугаарт, toDTO() арга ашиглан клиент рүү буцааж болох аюулгүй өгөгдлийг л сонгон буцаана. Энэ нь хэрэглэгчийн нууц үг зэрэг мэдрэмтгий талбарыг санамсаргүйгээр задруулахаас сэргийлнэ.

Model нь бизнес логик агуулдаггүй, Service эсвэл Repository-г дуудаж болохгүй. Зөвхөн өгөгдлийн бүтэц болон хариуны форматтай холбоотой ажлыг л хариуцна.

---

### 3.2 Repository (Репозитори)

Repository нь өгөгдлийн санд хандах бүх үйлдлийг нэг дор төвлөрүүлэх давхарга юм. Энэ давхарга нь mysql2 сангийн pool ашиглан SQL query бичиж гүйцэтгэнэ. Өгөгдлийн санд хандах бүх код зөвхөн энэ давхаргад байх ёстой бөгөөд Service эсвэл Controller-д шууд SQL бичих нь буруу хандлага юм.

Repository-ийн гол үүрэг нь өгөгдөл унших, бичих, засах, устгах үйлдлүүдийг гүйцэтгэх явдал юм. findAll арга нь хуудаслалт болон шүүлтийн параметрүүдийг хүлээн авч, WHERE нөхцөл динамикаар үүсгэн жагсаалт буцаана. findById арга нь тухайн ID-тай мөрийг хайж, байвал Model объект болгон буцаана, байхгүй бол null буцаана. create арга нь шинэ мөр оруулаад нэмэгдсэн мөрийн ID-аар дахин findById дуудан бүрэн объект буцаана. update арга нь тухайн мөрийг засаад шинэчлэгдсэн мөрийг буцаана. softDelete арга нь мөрийг физиклээр устгахын оронд deleted_at талбарт одоогийн цагийг тэмдэглэж "устгагдсан" гэж тэмдэглэнэ. Ийм байдлаар устгасан өгөгдлийг хожим нь сэргээх, эсвэл аудит хийх боломжтой болдог.

Repository нь бизнес дүрэм агуулж болохгүй. Жишээлбэл нэрийн давхцлыг Repository биш Service шалгах ёстой. Repository нь зөвхөн "өгөгдлийн санд юу хийх вэ" гэдгийг л мэдэх ёстой.

Repository-г ашиглахын давуу тал нь ирээдүйд өгөгдлийн сангаа сольсон тохиолдолд (жишээ нь MySQL-ээс PostgreSQL руу шилжих) зөвхөн Repository-ийн файлыг өөрчлөхөд л хангалттай болдог. Service болон Controller огт өөрчлөгдөх шаардлагагүй.

---

### 3.3 Service (Үйлчилгээ)

Service нь бизнес логикийг агуулах давхарга юм. Controller болон Repository-ийн хоорондох зуучлагч үүрэгтэй. Хэрэглэгчийн хүсэлтийг биелүүлэхийн тулд ямар дүрэм шалгах, ямар дарааллаар Repository-г дуудах гэдгийг Service шийднэ.

Service-ийн гол үүрэг нь бизнес дүрэм шалгах явдал юм. Жишээлбэл шинэ бүтээгдэхүүн үүсгэхийн өмнө ижил нэртэй бүтээгдэхүүн аль хэдийн байгаа эсэхийг шалгана. Байгаа бол ConflictError шидэж, Controller-д мэдэгдэнэ. Тодорхой бүтээгдэхүүнийг харахын өмнө тухайн ID байгаа эсэхийг шалгана. Байхгүй бол NotFoundError шидэнэ. Service нь нэг үйлдлийн хүрээнд олон Repository дуудаж болно. Жишээлбэл захиалга үүсгэхдээ бүтээгдэхүүний нөөцийг шалгах, захиалга хадгалах, нөөцийг бууруулах гурван Repository-г дараалан дуудаж болно.

Service нь HTTP хүсэлт, хариуны объектуудтай огт харилцахгүй. req, res зэрэг Express-ийн объектуудыг Service мэдэх шаардлагагүй. Service нь цэвэр JavaScript логик байх ёстой.

---

### 3.4 Controller (Хянагч)

Controller нь HTTP давхарга болон бизнес давхаргын хоорондох гүүр үүрэгтэй. Express Router-ийн callback функцүүд энд байрладаг. Хүсэлт ирэхэд Controller хүлээн авч, шаардлагатай параметрүүдийг уншиж, Service-г дуудаад, хариуг JSON форматаар буцаана.

Controller-д бизнес логик байж болохгүй. SQL query бичих, нэрийн давхцлыг шалгах зэрэг бизнесийн шийдвэр гаргах ажлыг Controller биш Service хийх ёстой. Controller нь зөвхөн хүсэлтийг хүлээн авч, Service-д дамжуулж, хариуг форматлан буцаах ажлыг л хариуцна. Алдаа гарвал next(err) дуудан Global Error Handler-т шилжүүлнэ.

---

### 3.5 Routes (Маршрут)

Routes нь URL endpoint болон Controller функцийн хоорондох холбоосыг тодорхойлно. Ямар HTTP method (GET, POST, PUT, DELETE) болон ямар URL нь аль Controller функцтэй холбогдохыг энд зарлана. Мөн аль endpoint нь нэвтрэлт шаардах, аль нь ямар роль шаардах гэдгийг middleware хэлбэрээр энд тодорхойлно.

---

### 3.6 Middleware (Завсрын боловсруулалт)

Middleware нь хүсэлт Controller-д хүрэхийн өмнө дундуур нь дамжих боловсруулалтын үе шат юм. authenticate middleware нь Authorization header-аас JWT токен уншиж, хүчинтэй эсэхийг шалгаад, хэрэглэгчийн мэдээллийг req.user-д тохируулна. authorize middleware нь req.user.role-ийг шалгаж, шаардлагатай роль байгаа эсэхийг баталгаажуулна. errorHandler middleware нь Service болон Repository-с шидэгдсэн алдааг барьж, стандарт JSON форматаар клиент рүү буцаана.

---

## 4. Нэгдмэл Хариуны Бүтэц

Бүх endpoint нэгдмэл форматтай JSON хариу буцаана. Амжилттай тохиолдолд success талбар true утгатай, data талбарт хариуны өгөгдөл байна. Алдааны тохиолдолд success талбар false утгатай, errors талбарт алдааны дэлгэрэнгүй мэдээлэл байна. Хариу бүрт message, timestamp талбарууд заавал байна.

---

## 5. JWT Authentication

Хэрэглэгч нэвтрэхдээ POST /api/v1/auth/login руу имэйл болон нууц үгээ илгээнэ. Server нь нууц үгийг bcryptjs ашиглан шалгаж, зөв бол jsonwebtoken ашиглан accessToken үүсгэж буцаана. Дараагийн хүсэлтүүдэд хэрэглэгч Authorization: Bearer токен хэлбэрээр header-т токен дамжуулна. authenticate middleware нь токен хүчинтэй эсэхийг шалгаж, хэрэглэгчийн мэдээллийг req.user-д хадгална. authorize middleware нь тухайн хэрэглэгч шаардлагатай роль эзэмшиж байгаа эсэхийг шалгана.

---

## 6. Pagination

Жагсаалт буцаадаг бүх endpoint хуудаслалтыг дэмжих ёстой. Хүсэлтэд page болон size параметрүүдийг query string-ээр дамжуулна. Repository-д LIMIT болон OFFSET ашиглан тухайн хуудасны мөрүүдийг авч, нийт мөрийн тоог COUNT query-гаар тусад нь авна. Хариунд data, total, page, size, totalPages талбарууд байна.

---

## 7. Давхаргын Хоорондох Дүрэм

| Дүрэм | Тайлбар |
|---|---|
| Controller → Service | Controller нь зөвхөн Service-ийг дуудна |
| Service → Repository | Service нь Repository-г дуудна |
| Repository → DB | Repository нь mysql2 pool ашиглана |
| Controller-д логик байхгүй | Бизнес логик зөвхөн Service-д |
| Repository-д логик байхгүй | Бизнес дүрэм зөвхөн Service-д |
| Model нь Service-г дуудахгүй | Доод давхарга дээд давхарга мэдэхгүй |
| toDTO() ашиглах | Model-ийн нууц талбарыг нуух |
| SQL зөвхөн Repository-д | Бусад давхаргад SQL бичихгүй |

---

## 8. Нийтлэг Алдаанаас Зайлсхийх

Controller-д SQL query бичих нь бусад хэсэгт дахин ашиглах боломжгүй болгодог тул буруу хандлага юм. Controller нь зөвхөн Service дуудаж, хариу буцаах ёстой.

Repository-д бизнес дүрэм бичих нь давхаргын хариуцлагыг будлиулдаг. Нэрийн давхцал шалгах, утга баталгаажуулах зэрэг бизнесийн шийдвэрийг Service-д байлгах ёстой.

Model-ийг шууд буцаах нь нууц үг зэрэг мэдрэмтгий талбаруудыг задруулах эрсдэлтэй. Заавал toDTO() ашиглан зөвшөөрөгдсөн талбаруудыг л буцаах хэрэгтэй.

## 9. HTTP Status Code Лавлах

| Код | Утга | Ашиглах үе |
|---|---|---|
| 200 | OK | Амжилттай унших, засах |
| 201 | Created | Амжилттай үүсгэх |
| 204 | No Content | Амжилттай устгах |
| 400 | Bad Request | Validation алдаа |
| 401 | Unauthorized | Нэвтрээгүй |
| 403 | Forbidden | Эрхгүй |
| 404 | Not Found | Олдсонгүй |
| 409 | Conflict | Давхцал |
| 500 | Internal Server Error | Серверийн алдаа |
