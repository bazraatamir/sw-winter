# Container Networking — Дэлгэрэнгүй Онолын Гарын Авлага

---

## 1. Асуудал: Контейнерүүд Хэрхэн Харилцах Вэ?

Нэг контейнер тусгаарлагдсан орчинд ажилладаг гэдгийг бид мэднэ. Гэтэл Task Manager-т MySQL контейнер болон Task Service контейнер хоёр нэг нэгтэйгээ харилцах шаардлагатай. Хэрэглэгч гаднаас Task Service-д хандах шаардлагатай. Notification Service RabbitMQ-тай харилцах шаардлагатай.

Тусгаарлагдсан контейнерүүд хэрхэн харилцаж, гаднаас хэрхэн хандаж болох вэ? Энэ асуултад **Container Networking** хариулдаг.

---

## 2. Docker Network гэж юу вэ?

Docker Network бол контейнерүүдийг хооронд нь болон гадаад ертөнцтэй **холбодог виртуал сүлжээ** юм. Бодит компьютерийн сүлжээтэй адил зарчмаар ажилладаг — IP хаяг, DNS, port, routing бүгд байдаг, зөвхөн виртуал орчинд.

Зүйрлэл: Оффисийн дотоод сүлжээ гэж бодоорой. Нэг оффис дотрын компьютерууд дотоод IP-ээр чөлөөтэй харилцаж болно. Гаднаас ирэх хүн зөвхөн нийтийн IP болон тодорхой port-оор хандаж болно. Docker Network яг ийм оффисийн дотоод сүлжээний үүрэг гүйцэтгэнэ.

```
Гадаад ертөнц
      │
      │ Port mapping-аар
      ↓
┌─────────────────────────────────────────┐
│           Docker Network                │
│                                         │
│  [MySQL]  [RabbitMQ]  [task-service]   │
│    │           │            │           │
│    └───────────┴────────────┘           │
│         Дотооод DNS-ээр харилцана       │
└─────────────────────────────────────────┘
```

---

## 3. Docker Network Driver-ийн Төрлүүд

Docker нь хэд хэдэн network driver-тэй. Driver нь сүлжээ хэрхэн ажиллахыг тодорхойлдог.

### 3.1 Bridge — Хамгийн Түгээмэл

Bridge network бол Docker-ийн **өгөгдмөл сүлжээний хэлбэр** юм. Docker Compose-д `networks` хэсгийг заагаагүй бол бүх сервис нэг bridge network-т автоматаар орно.

Bridge гэдэг нь утгаараа "гүүр" — тус тусдаа ажилладаг контейнерүүдийг нэг виртуал сүлжээнд нэгтгэдэг дундын цэг юм. Docker энэ гүүрийг `docker0` нэртэй виртуал интерфейс болгон хостын дотор үүсгэдэг. Контейнер бүр энэ гүүрт холбогдоход дотоод IP хаяг (`172.17.0.x`) автоматаар оноогддог.

```
Хост компьютер
┌──────────────────────────────────────────────┐
│                                              │
│  docker0 (bridge: 172.17.0.1)               │
│       │                                      │
│       ├── Container A: 172.17.0.2            │
│       ├── Container B: 172.17.0.3            │
│       └── Container C: 172.17.0.4            │
│                                              │
└──────────────────────────────────────────────┘
```

Bridge дотрох контейнерүүд хоорондоо **нэрээрээ** харилцаж болно — Docker-ийн дотоод DNS нэрийг IP болгон автоматаар хувиргадаг. Гаднаас хандахын тулд заавал `ports` дамжуулан port mapping хийх шаардлагатай.

Docker Compose ашиглах үед `networks` тодорхойлоогүй бол цаана нь bridge network автоматаар үүсч сервис бүр нэгэн зэрэг орно. Хэрэв `networks` тодорхойлвол Docker Compose тусдаа bridge-үүд үүсгэж аль сервис аль bridge-д байхыг тодорхой хянаж болно — аюулгүй байдал нэмэгдэнэ.

### 3.2 Host — Хостын Сүлжээг Шууд Ашиглах

Host network ашиглах үед контейнер **виртуал сүлжээний давхарга огт ашиглахгүй** — хостын сүлжээний интерфейсийг шууд ашиглана. Энэ тохиолдолд контейнер нь хостын нэг хэсэг болсон мэт ажилладаг.

Үр дүнд нь контейнерийн IP хаяг нь хостын IP хаяг болно, нээсэн port нь хостын port болно, port mapping хэрэггүй болно.

```
Хост: 192.168.1.10
Container (host network):
  → IP мөн л 192.168.1.10
  → PORT 3002 нээвэл хостын 3002 шууд
  → port mapping хэрэггүй
```

Виртуал сүлжээний давхарга байхгүй тул **хамгийн хурдан** хэлбэр боловч тусгаарлалт бүрэн байхгүй гэдгийг анхаарах хэрэгтэй. Container хостын бүх сүлжээний интерфейсийг харж болно, хостын port-уудтай зөрчилдөж болно. Нэмж хэлэхэд зөвхөн **Linux** дээр бүрэн ажилладаг — Windows болон Mac дээр Docker Desktop виртуал машин дамжуулан ажилладаг тул host network-ийн давуу тал алдагддаг.

Хэрэглэх нөхцөл: сүлжээний хурд шаардлагатай, тусгаарлалт хоёрдогч чухал тохиолдолд.

### 3.3 None — Сүлжээгүй Бүрэн Тусгаарлалт

None network бол нэрнээсээ тодорхой — контейнер **ямар ч сүлжээнд холбогддоггүй**. Дотоод loopback хаяг (`127.0.0.1`) л байна, өөр юу ч байхгүй.

```
Container (none network):
  loopback: 127.0.0.1   ← дотооддоо л харилцаж болно
  Гаднаас хандах        ← боломжгүй
  Бусад container       ← харилцах боломжгүй
  Интернет              ← хандах боломжгүй
```

Энэ хэлбэр нь аюулгүй байдлыг хамгийн дээд түвшинд баталгаажуулдаг. Гаднаас ямар ч хандалт байхгүй тул сүлжээний халдлагаас бүрэн хамгаалагдсан байдаг. Харин ийм контейнер өөрийн дотоод процессаасаа гадна юутай ч харилцаж чадахгүй.

Хэрэглэх нөхцөл: зөвхөн тооцоолол эсвэл файл боловсруулах ажил хийх контейнер, сүлжээний холболт огт шаардлагагүй, аюулгүй байдал хамгийн чухал нөхцөл.

### 3.4 Overlay — Олон Хост Хооронд

Хэд хэдэн Docker хост (физик сервер) дээрх контейнерүүдийг **нэг виртуал сүлжээнд** холбоно. Docker Swarm болон Kubernetes-т ашиглагддаг.

```
Сервер 1                  Сервер 2
┌──────────────┐           ┌──────────────┐
│ [Container A]│           │ [Container B]│
│              │──overlay──│              │
└──────────────┘   сүлжээ  └──────────────┘
```

Хэрэглэх нөхцөл: production дахь cluster орчин, олон сервер дээр тархсан систем.

## 4. Container DNS — Нэрээр Хандах

Docker Compose-ийн хамгийн хэрэгтэй онцлогуудын нэг бол **дотоод DNS** юм. Нэг Network дотрох контейнерүүд IP хаяг биш **нэрээрээ** харилцаж болно.

```yaml
services:
  mysql:         ← "mysql" гэсэн DNS нэр
    image: mysql:8.0

  task-service:
    environment:
      DATABASE_URL: mysql://root:pass@mysql:3306/db
      #                              ↑
      #               IP биш "mysql" нэрээр хандана
```

Docker Compose сервис бүрт тухайн сервисийн нэртэй тохирох DNS бичлэг автоматаар үүсгэдэг. `task-service` дотроос `mysql` гэж хандахад Docker-ийн дотоод DNS `mysql` container-ийн IP-г автоматаар олж өгдөг.

```
task-service container:
  "mysql" гэж хандана
       ↓
  Docker DNS: mysql → 172.20.0.2
       ↓
  MySQL container (172.20.0.2)
```

Энэ нь маш чухал давуу тал — Container дахин эхлэх бүрт IP хаяг өөрчлөгдөж болно, гэхдээ нэр хэвээр байдаг тул холболт тасрахгүй.

## 5. Port Mapping — Гаднаас Хандах

Container дотроо port нээдэг боловч хостын гаднаас тухайн port-д хандахын тулд **port mapping** хэрэгтэй.

```
Хэрэглэгч → Хостын 3002 → Container-ийн 3002
              Port mapping
```

```yaml
services:
  task-service:
    ports:
      - "3002:3002"    # "хостын_port:container-ийн_port"
```

Port mapping-ийн янз бүрийн хэлбэрүүд:

```yaml
ports:
  - "3002:3002"              # хостын 3002 → container-ийн 3002
  - "8080:3002"              # хостын 8080 → container-ийн 3002 (өөр port)
  - "127.0.0.1:3002:3002"   # зөвхөн localhost-аас (гадаад IP-ийн хандалт хаагдсан)
  - "3002"                   # хостын санамсаргүй port → container-ийн 3002
```

### Port Mapping байхгүй бол

```yaml
services:
  mysql:
    image: mysql:8.0
    # ports заагаагүй
    # → Зөвхөн Docker Network дотроос хандаж болно
    # → Хостоос, гаднаас хандаж болохгүй
```

Аюулгүй байдлын хувьд дотоод сервисүүдэд (MySQL, RabbitMQ) port mapping хязгаарлах нь зүйтэй — зөвхөн шаардлагатай сервисүүдийн port-г нээнэ.

## 6. Network Тусгаарлалт — Аюулгүй Байдал

Бүх container нэг network-д байвал бие биенд хандаж болно. Тусдаа network үүсгэснээр тусгаарлалт хангадаг.

```yaml
services:
  mysql:
    networks:
      - backend        # Зөвхөн backend сүлжээнд

  rabbitmq:
    networks:
      - backend

  task-service:
    networks:
      - backend        # MySQL, RabbitMQ-тэй харилцана
      - frontend       # Гаднаас хүсэлт хүлээнэ

  notification-service:
    networks:
      - backend        # Зөвхөн RabbitMQ-тэй харилцана
                       # frontend сүлжээнд байхгүй тул гаднаас хандаж болохгүй

networks:
  backend:
  frontend:
```

```
Гадаад ертөнц
      │
      │ 3002 port-оор
      ↓
┌─── frontend network ───────────────────┐
│         [task-service]                  │
└─────────────┬──────────────────────────┘
              │
┌─── backend network ────────────────────┐
│  [mysql] [rabbitmq] [task-service]     │
│                     [notification]     │
└────────────────────────────────────────┘
```

`notification-service` нь `frontend` network-д байхгүй тул гаднаас шууд хандаж болохгүй. Зөвхөн `backend` network-оор RabbitMQ-тай харилцана.

## 7. Network Командууд

```bash
# ── Network жагсаалт харах ─────────────────────────────
docker network ls

# ── Network үүсгэх ─────────────────────────────────────
docker network create my-network
docker network create --driver bridge my-bridge

# ── Network-ийн дэлгэрэнгүй мэдээлэл ──────────────────
# Ямар container холбогдсон, IP хаяг гэх мэт
docker network inspect my-network

# ── Container-г network-д нэмэх ────────────────────────
docker network connect my-network task-service

# ── Container-г network-аас гаргах ─────────────────────
docker network disconnect my-network task-service

# ── Network устгах ─────────────────────────────────────
docker network rm my-network

# ── Ашиглагдаагүй network бүгдийг устгах ───────────────
docker network prune
```

## 8. Контейнер Дотроос Сүлжээ Шалгах

Контейнер дотроос сүлжээний холболт шалгах нь debug хийхэд маш хэрэгтэй.

```bash
# Container-т нэвтрэх
docker compose exec task-service sh

# Доторх контейнерт ping хийх
ping mysql
ping rabbitmq

# DNS шийдвэрлэлт шалгах
nslookup mysql
# Server: 127.0.0.11  ← Docker-ийн дотоод DNS сервер
# Address: 172.20.0.2  ← MySQL container-ийн IP

# Port нээлттэй эсэх шалгах
nc -zv mysql 3306
# Connection to mysql 3306 port [tcp/mysql] succeeded!

# wget-ээр шалгах
wget -qO- http://task-service:3002/health
```

## 9. Task Manager дахь Network Урсгал

Конкрет жишээгээр бүрэн урсгалыг харцгаая:

```
1. Хэрэглэгч гаднаас Task үүсгэх хүсэлт илгээнэ:
   http://localhost:3002/api/workspaces/1/tasks

        ↓ Port mapping: хостын 3002 → task-service container-ийн 3002

2. task-service container хүсэлт хүлээн авна

        ↓ Backend network дотоод DNS

3. task-service → mysql:3306 (Prisma)
   "mysql" нэр → 172.20.0.2 IP → MySQL container

        ↓ Task хадгалагдана

4. task-service → rabbitmq:5672 (amqplib)
   "rabbitmq" нэр → 172.20.0.3 IP → RabbitMQ container

        ↓ "task.created" event дамжуулагдана

5. notification-service ← rabbitmq:5672
   "rabbitmq" нэр → 172.20.0.3 IP → RabbitMQ container

        ↓ Event боловсруулна, мэдэгдэл илгээнэ

6. task-service → 201 хариу → Хэрэглэгч
```

## Товч Дүгнэлт

Container Networking бол контейнерүүдийг хооронд болон гадаад ертөнцтэй холбодог виртуал сүлжээ юм. Санах ойд байлгах зүйл:

- **Bridge** → нэг хост дотрын контейнерүүд — хамгийн түгээмэл, Docker Compose-ийн өгөгдмөл
- **Host** → хостын сүлжээ хуваалцана — хурдан, тусгаарлалтгүй
- **None** → сүлжээгүй — бүрэн тусгаарлагдсан
- **Overlay** → олон хост хооронд — Kubernetes, Swarm
- **Дотоод DNS** → Container нэрээр хандана, IP биш
- **Port mapping** → `"хост:container"` — гаднаас хандахын тулд
- **Network тусгаарлалт** → frontend/backend сүлжээ ялгах нь аюулгүй байдлыг нэмэгдүүлнэ
