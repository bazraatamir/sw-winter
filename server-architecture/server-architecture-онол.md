# Серверийн Архитектур — Дэлгэрэнгүй Онолын Гарын Авлага

---

## 1. Серверийн Архитектур гэж юу вэ?

Серверийн архитектур бол аппликейшний **бүтэц, зохион байгуулалт** юм — код хэрхэн хуваагдаж, сервисүүд хэрхэн харилцаж, өгөгдөл хэрхэн урсдагийг тодорхойлдог.

Зүйрлэл: Байшин барихын өмнө **зураг төсөл** хийдэг шиг, апп хөгжүүлэхийн өмнө архитектураа тодорхойлдог. Зураг төслөөс хамааран байшин ганц өрөө орон байр байж болно, эсвэл олон давхар оффис байшин байж болно.

Архитектур нь дараах асуултад хариулдаг:
- Кодыг хэрхэн **хуваах** вэ?
- Хэсгүүд хэрхэн **харилцах** вэ?
- Системийг хэрхэн **өргөтгөх** вэ?
- Алдаа гарвал **бусадт нөлөөлөх** үү?

---

## 2. Monolith Архитектур

### Гэж юу вэ?

Monolith буюу "нэгдмэл" архитектур бол аппликейшний бүх хэсэг **нэг кодын санд, нэг процесст** ажилладаг загвар юм.

```
┌─────────────────────────────────────────┐
│           НЭГДМЭЛ АППЛКЕЙШН            │
│                                         │
│  Auth логик                             │
│  Task логик                             │
│  Workspace логик      Нэг процесс       │
│  Notification логик                     │
│  File логик                             │
│                                         │
└─────────────────────────────────────────┘
              ↓
        Нэг DB сервер
```

Таны Task Manager одоо яг ийм — нэг Express апп дотор бүх логик байна.

### Дотоод бүтэц

Monolith нь "нэг цул блок" биш — дотроо **давхаргуудад** хуваагддаг:

```
┌─────────────────────────────┐
│         Routes              │  ← HTTP хүсэлт хүлээн авна
├─────────────────────────────┤
│       Controllers           │  ← Логик удирдана
├─────────────────────────────┤
│         Models              │  ← Өгөгдөлтэй ажиллана
├─────────────────────────────┤
│        Database             │  ← MySQL
└─────────────────────────────┘
```

Энэ нь таны Task Manager-ийн MVC бүтэц юм.

### Давуу талууд

**Хөгжүүлэхэд хялбар.** Бүх код нэг газарт байдаг тул шинэ хөгжүүлэгч хурдан ойлгодог. Local дээр ажиллуулахад зөвхөн нэг процесс эхлүүлэхэд хангалттай.

**Debug хялбар.** Stack trace нэг аппын дотор бүгд харагдана. Хаана алдаа гарсаныг олоход хялбар.

**Deploy энгийн.** Нэг апп, нэг сервер — нэг тушаалаар deploy хийнэ.

**Function дуудалт хурдан.** Хэсгүүд нэг процесст байдаг тул сүлжээний overhead байхгүй — auth логик task логикийг шууд дуудна.

### Сул талууд

**Нэг хэсэг унавал бүгд унана.** Notification сервис алдаатай бол Task сервис ч ажиллахгүй болно.

```
Notification логик алдаа → Бүх апп зогсоно 😱
```

**Масштаблах хэцүү.** Зөвхөн Task логик их ачаалалтай ч бүх аппыг хуулах шаардлагатай болно.

```
Хэрэгтэй: Task логик × 3
Бодит байдал: Бүх апп × 3 (Auth, Workspace, Notification ч хамт)
              → Нөөц дэмий зарцуулагдана
```

**Кодын хэмжээ өсөх тусам удаашрана.** Мянган файлтай болоход build удааширч, ойлгоход хэцүү болно.

**Технологи сонгох хязгаарлагдмал.** Нэг хэсэгт Python ашиглахыг хүссэн ч бүх апп Node.js байх ёстой.

### Хэзээ Monolith сонгох вэ?

```
✅ Апп шинэ, багийн гишүүн цөөн
✅ Хэрэглэгч цөөн, ачаалал бага
✅ Бизнесийн логик тодорхойгүй байгаа
✅ Хурдан MVP гаргах шаардлагатай
```

## 3. Microservice Архитектур

### Гэж юу вэ?

Microservice архитектур бол аппликейшнийг **жижиг, бие даасан сервисүүдэд** хуваах загвар юм. Сервис бүр өөрийн гэсэн зорилготой, тусдаа ажилладаг.

```
┌──────────┐  ┌──────────┐  ┌──────────────┐
│  Auth    │  │  Task    │  │  Workspace   │
│ Service  │  │ Service  │  │   Service    │
│ :3001    │  │ :3002    │  │   :3003      │
└────┬─────┘  └────┬─────┘  └──────┬───────┘
     │              │               │
  MySQL DB      MySQL DB        MySQL DB
  (users)       (tasks)        (workspaces)
```

**Гол зарчим:** Сервис бүр өөрийн **тусдаа DB**-тэй — нэг сервис нөгөөгийн DB-д шууд хандахгүй.

### Сервисүүд хэрхэн харилцдаг вэ?

**Синхрон (HTTP/REST):** Нэг сервис нөгөөгөөс шууд хариу хүлээнэ.

```
Task Service → HTTP хүсэлт → Auth Service
                           ← Хариу (хэрэглэгч байна уу?)
```

**Асинхрон (Message Queue):** Нэг сервис мессеж явуулаад хариу хүлээхгүй — нөгөө сервис цагтаа уншина.

```
Task Service → [RabbitMQ/Kafka] → Notification Service
              мессеж үлдээнэ       цагтаа уншиж имэйл илгээнэ
```

### Task Manager-ийн Microservice загвар

```
                    [API Gateway]
                         │
        ┌────────────────┼────────────────┐
        ↓                ↓                ↓
  ┌──────────┐    ┌──────────┐    ┌──────────────┐
  │  Auth    │    │  Task    │    │  Workspace   │
  │ Service  │    │ Service  │    │   Service    │
  └────┬─────┘    └────┬─────┘    └──────┬───────┘
       │               │                  │
   users DB        tasks DB          workspaces DB

        Task үүсгэгдэнэ
               ↓
        [Message Queue]
               ↓
        ┌──────────────┐
        │ Notification │
        │   Service    │
        └──────────────┘
```

### Давуу талууд

**Сервис бие даасан.** Notification сервис унасан ч Task сервис ажиллсаар байна.

```
Notification унана → Task сервис хэвийн ажиллана ✓
```

**Тусдаа масштаблах.** Зөвхөн их ачаалалтай сервисийг нэмнэ.

```
Task сервис их ачаалалтай → Task сервисийг × 3 болгоно
Auth, Workspace сервис хэвийн → нэмэхгүй
→ Нөөц хэмнэнэ
```

**Технологи чөлөөтэй сонгох.** Auth сервис Node.js, аналитик сервис Python байж болно.

**Багуудаар хуваарилах.** Нэг баг Task сервисийг, нөгөө баг Workspace сервисийг тус тусад нь хөгжүүлнэ — бие биедээ саад болохгүй.

### Сул талууд

**Нарийн төвөгтэй.** Нэг хүсэлт хэд хэдэн сервисийг дамжих тул debug хэцүү болно.

```
Алдаа хаана гарсан бэ?
API Gateway? Auth? Task? DB? Message Queue? 🤔
```

**Сүлжээний overhead.** Function дуудалтын оронд HTTP хүсэлт явуулах тул удаашрал нэмэгдэнэ.

**Distributed transaction хэцүү.** Хэд хэдэн сервисийн DB-г нэгэн зэрэг шинэчлэх нь monolith-аас хамаагүй хүндрэлтэй.

```
Task үүсгэх + Notification илгээх + Log бичих
→ Гурван өөр DB, гурван өөр сервис
→ Нэг нь амжилтгүй бол бусдыг буцаах хэрхэн? 🤔
```

**Дэд бүтэц их шаардана.** Сервис бүрд тусдаа deploy, тусдаа monitoring, тусдаа log хэрэгтэй.

### Хэзээ Microservice сонгох вэ?

```
✅ Апп том, баг олон (50+ хөгжүүлэгч)
✅ Хэсгүүдийн ачаалал тэс өөр
✅ Хурдан, тогтвортой deploy шаардлагатай
✅ Бизнесийн логик тодорхой, тогтворжсон
```

## 4. Monolith vs Microservice

```
                MONOLITH          MICROSERVICE
                   │                   │
Нарийн төвөгтэй   Бага               Их
Deploy             Хялбар             Хүндрэлтэй
Масштаблах         Бүгдийг нэгэн зэрэг  Тусдаа
Алдааны тархалт   Нэг унавал бүгд    Тусгаарлагдсан
Хөгжүүлэлт        Хурдан (эхэндээ)   Удаан (эхэндээ)
Баг хэмжээ        Цөөн               Олон
```

### Бодит компаниудын жишээ

**Netflix:** Эхлээд Monolith байсан → 2009 онд Microservice руу шилжсэн. Одоо 700 гаруй microservice ажиллуулдаг.

**Amazon:** "Two-pizza team" зарчим — нэг баг хоёр пицца идэж чадах хэмжээний (6–8 хүн) байх ёстой. Баг бүр өөрийн сервисийг эзэмшдэг.

**Uber:** Эхлээд нэг Monolith → хурдан өссөний дараа Microservice руу шилжсэн.

## 5. Modulith — Дунд Зам

Monolith ба Microservice хоёрын **хооронд** орших дунд зам бий. Үүнийг **Modulith** (Module + Monolith) гэнэ.

```
┌─────────────────────────────────────────┐
│           НЭГДМЭЛ АППЛКЕЙШН            │
│  ┌─────────┐  ┌─────────┐  ┌────────┐  │
│  │  Auth   │  │  Task   │  │  Work  │  │
│  │ Module  │  │ Module  │  │ Module │  │
│  └─────────┘  └─────────┘  └────────┘  │
│       Модуль бүр тусдаа, нэг процессд  │
└─────────────────────────────────────────┘
              ↓
        Нэг DB (тусдаа schema)
```

**Давуу тал:**
- Monolith шиг **deploy хялбар**
- Microservice шиг **код тусгаарлагдсан**
- Ирээдүйд Microservice руу **шилжихэд бэлэн**

## 6. Task Manager-ийн Архитектурын Зөвлөмж

Таны Task Manager одоо **Monolith** бөгөөд энэ нь зөв сонголт. Яагаад гэвэл:

```
Одоогийн байдал:
✓ Нэг хөгжүүлэгч
✓ Суралцах зорилготой
✓ Хэрэглэгч цөөн
✓ Логик тодорхойгүй байгаа

→ Monolith хамгийн тохиромжтой
```

Гэхдээ дотоод бүтцийг **Modulith загвараар** зохион байгуулбал ирээдүйд Microservice руу шилжих хялбар болно:

```
task-manager/
├── modules/
│   ├── auth/
│   │   ├── auth.routes.js
│   │   ├── auth.controller.js
│   │   └── auth.model.js
│   ├── tasks/
│   │   ├── task.routes.js
│   │   ├── task.controller.js
│   │   └── task.model.js
│   └── workspaces/
│       ├── workspace.routes.js
│       ├── workspace.controller.js
│       └── workspace.model.js
├── shared/
│   ├── middleware/
│   ├── utils/
│   └── config/
└── app.js
```

Модуль бүр **бие даасан** — нэг модулийн өөрчлөлт нөгөөд нөлөөлөхгүй. Ирээдүйд тусдаа сервис болгоход хялбар.

---

## Товч дүгнэлт

| | Monolith | Modulith | Microservice |
|---|---|---|---|
| **Хэрэглэгдэх үед** | Жижиг апп, MVP | Дунд хэмжээний апп | Том апп, олон баг |
| **Нарийн төвөгтэй** | Бага | Дунд | Их |
| **Масштаблах** | Бүгдийг нэгэн зэрэг | Бүгдийг нэгэн зэрэг | Тусдаа |
| **Эхлэхэд** | Хурдан | Дунд | Удаан |

**Алтан дүрэм:** Microservice-ээс эхлэхгүй — эхлээд Monolith/Modulith хийж, **хэрэгцээ гарсан үед** л Microservice руу шилж.
